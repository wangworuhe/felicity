# Feature Bug Review

按以下流程执行，目标是快速给出“可落地、可复现、可修复”的诊断结论。

## 执行原则

- 聚焦功能正确性与用户体验异常，不做纯风格审查。
- 所有结论必须有代码证据，禁止无证据猜测。
- 默认不改代码，只输出分析与建议。
- 先复现场景，再定位根因；不能复现时明确阻塞信息。

## 标准流程

1. 明确问题定义
- 提炼：现象、触发动作、期望行为、实际行为、影响范围。
- 若信息不足，优先从仓库上下文补齐；仍不足时再提出最小澄清问题。

2. 建立排查范围
- 先查入口：页面/组件 `wxml` 事件绑定、路由、参数来源。
- 再查逻辑：`bindinput`/`bindtap`/生命周期/异步调用/状态更新。
- 最后查依赖：`services/`、`utils/`、云函数或接口返回结构。

3. 构建根因链路
- 至少形成一条完整链路：触发条件 -> 代码路径 -> 状态变化 -> 异常结果。
- 如有多个可疑点，按概率与影响排序，并给出主因/次因。

4. 输出双模板报告
- 先输出“给人看的简版摘要”（帮助快速理解问题）。
- 再输出“给 AI Agent 的 YAML 交接报告”。
- 模板结构见 `references/report-template.md`。

## 必查清单

- 状态管理：是否存在受控输入抖动、整对象回写、无差量更新。
- 事件模型：事件绑定是否错误、dataset 参数是否稳定。
- 渲染路径：频繁 `setData` 是否触发重绘副作用。
- 异步时序：并发请求覆盖、旧响应回写新状态、竞态条件。
- 边界条件：空值、极值、重复提交、快速连点、分页边界。
- 数据契约：字段命名与结构是否前后端一致。
- 平台特性：微信小程序输入组件、滚动、焦点、授权等行为差异。

## 输出要求（双模板）

- 必须按顺序输出两段：
  1) 人类可读简版摘要（Markdown）
  2) AI Agent 交接 YAML（单个 YAML 代码块）
- YAML 字段结构参照 `references/report-template.md`。
- 严重级别使用小写：`critical` / `high` / `medium` / `low`。
- 证据必须带绝对路径与行号。
- 修复建议至少包含 `preferred_plan` 与 `alternative_plan`。
- 必须包含 `decision_gate_for_next_agent`：
  - 先核验“报告是否属实、方案是否合理”；
  - 若属实且合理，可直接修复；
  - 若不属实或不合理，先提出异议并等待用户确认。
- 默认不输出补丁；仅当用户明确要求当前 Agent 修复时才改代码。

## 示例触发语句

- “帮我查这个功能为什么不对”
- “这里有 bug，帮我找根因”
- “做一次功能代码审查，不要改代码”
- “输入框光标乱跳，查原因并给修复建议”
